<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TUI/player token -> RetconService | User input authorizes canon removal; must be exact and turn-specific. |
| RetconService -> graph checkpoint history | Rollback depends on checkpoint integrity and must block if unavailable. |
| RetconService -> vault/player projection | Repair output may touch player-visible vault files. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-04 | Spoofing | Confirmation token | mitigate | Require exact `RETCON {turn_id}` token and recompute preview at confirmation time. |
| T-08-05 | Tampering | Graph checkpoint rewind | mitigate | Use existing `_rewind_to_checkpoint` with persisted prior final checkpoint only; block if absent. |
| T-08-06 | Information Disclosure | Retcon completion/log note | mitigate | Completion message is spoiler-safe and references turn ids/counts, not removed canon details or GM-only text. |
| T-08-07 | Denial of Service | Derived rebuild | mitigate | If rebuild/sync fails, raise repair guidance; canonical SQL exclusion already protects future context. |
</threat_model>
