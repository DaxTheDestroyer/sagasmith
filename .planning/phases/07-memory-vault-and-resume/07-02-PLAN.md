<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Deterministic service → Filesystem (master vault) | Archivist turns in-memory page objects into durable on-disk markdown. Partial or corrupt writes break the source-of-truth contract. |
| SQLite commit → Vault write | Post-commit vault failures must not lose SQLite history, but must signal repair need. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-05 | Tampering | Vault write failure after SQLite commit leading to inconsistency | mitigate | If any vault page write fails, update turn_records.status to "needs_vault_repair"; user can run `ttrpg vault repair`. All writes before failure are rolled back via tempfile cleanup; no partial pages committed. |
| T-07-06 | Information Disclosure | Player-vault sync error leaves stale or corrupted player files | accept | Sync failures set a persistent `sync_warning` flag surfaced in TUI; player can re-run `ttrpg vault sync` manually (Plan 07-05). |
| T-07-07 | Denial of Service | Corrupt vault file blocks entire sync | mitigate | Atomic writes protect against corruption; individual page parse errors during sync are caught and reported per-page without stopping full sync (best-effort). |
| T-07-08 | Elevation of Privilege | Malicious campaign directory with symlink attacks on vault paths | mitigate | Vault paths are constructed inside known app-data directories; we avoid following symlinks outside campaign root by resolving realpath and ensuring it's within allowed base. Not fully enforced in MVP but low risk. |
</threat_model>
