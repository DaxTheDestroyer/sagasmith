<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer shell → local filesystem | Developer-run `uv`, `make`, and `sagasmith` CLI commands read/write project files. Limited attack surface at this phase because no network I/O, no secrets, no persistence, and no untrusted input. |
| git working tree → remote repository | Committed files (lockfile, configs) become the record developers depend on. Supply-chain and secrets-in-commits are the only plausible threats this phase unlocks. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | `uv.lock` | mitigate | Commit the lockfile; CI/developer workflows use `uv sync --frozen` (documented in later CI phase) so transitive dependency swaps require a reviewed lockfile change. |
| T-01-02 | Information Disclosure | `.pre-commit-config.yaml` + developer workflow | mitigate | `gitleaks` hook runs on every commit to catch accidentally staged secrets before they reach the remote, enforcing PROJECT.md constraint "API keys must never be persisted to campaign files, vaults, transcripts, checkpoints, or debug logs." |
| T-01-03 | Elevation of Privilege | Third-party ruff/pyright/gitleaks pre-commit repos | accept | Pins use explicit `rev:` SHAs/tags (not `main`) so a compromised upstream cannot silently inject hooks; risk at Phase 1 is low (no runtime secrets yet). Revisit when CI pipeline is introduced. |
