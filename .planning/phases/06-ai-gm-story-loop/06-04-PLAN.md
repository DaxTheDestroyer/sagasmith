# Task 4: Implement Scene Rendering with Safety Gates (Orator)

**Task ID:** 06-04
**Owner:** Kilo (AI Assistant)
**Phase:** 06-ai-gm-story-loop
**Requirements:** AI-07, AI-08, AI-09, AI-10, SAFE-01, SAFE-02, SAFE-03, D-01, D-02, D-04, D-05

## Overview

Replace the stub narration with LLM-generated prose that respects safety policies and mechanical outcomes. This is the primary player-facing component that delivers the narrative experience.

**Design constraint — read first:** the safety post-gate and "live streaming" cannot coexist. If a classifier might force a rewrite, the player cannot have already seen the original tokens. This task implements the **buffered stream-after-classify** pattern from `06-PLAN.md` D-06.1. The player sees a paced playback that *looks* like a live stream but is actually a validated buffer.

**Mechanical consistency** is enforced via the prompt-side constraint encoding + deterministic regex audit defined in D-06.2. No second LLM verifier in Phase 6.

### Step 3: Implement Scene Rendering Logic (buffered stream-after-classify)

**File:** `src/sagasmith/agents/orator/skills/scene-rendering/logic.py`

Implement the D-06.1 pipeline:

1. Build the prompt via `prompts/orator/scene_rendering.py` (per D-06.5). Encode `CheckResult` payloads as structured constraint tokens in the user prompt; include an explicit "do not contradict these mechanical outcomes" instruction.
2. Call `LLMClient.stream(...)`. Accumulate tokens into a private `bytearray`/`list[str]` buffer — **do not write to `pending_narration` yet**.
3. Run the **inline hard-limit pattern matcher** (Step 3.5) against the running buffer on each token. On a hit, cancel the stream early and jump to rewrite.
4. On stream completion, run the **post-gate classifier** (Step 4 + Task 7) against the buffered text.
5. Run the **mechanical-consistency audit** (Step 6) against the buffered text.
6. If all gates pass, **playback** the buffered tokens to `pending_narration` at a paced rate (e.g. 30–60 tokens/sec) so the TUI sees a stream-like UX. The "first validated token visible within 2 seconds p50" metric is measured from player-input receipt to first playback token.
7. If any gate fails, request a rewrite (up to two retries total). After two failures, emit fallback narration.
8. Handle network/timeout errors via `invoke_with_retry` patterns adapted for streaming.

### Step 3.5: Inline Hard-Limit Pattern Matcher

**File:** `src/sagasmith/services/safety_inline_matcher.py`

Cheap regex/keyword scanner over `ContentPolicy.hard_limits`:
- Compile patterns once at service init
- `match(buffer: str) -> Optional[Match]` returns first hit or None
- Designed to run on every accumulated token-window without measurable latency
- Used by the Orator to short-circuit doomed generations before paying for full output

### Step 4: Implement Dice UX Handling

**File:** `src/sagasmith/agents/orator/dice_ux.py`

Create logic for handling different dice UX modes:
- Auto: Weave outcomes seamlessly into narration
- Reveal: Narrate attempt, pause for dice overlay, resume with outcome
- Hidden: Never name rolls, DCs, or modifiers

### Step 5: Update Orator Node

**File:** `src/sagasmith/agents/orator/node.py`

Modify the orator_node function to:
- Replace stub narration with the buffered stream-after-classify pipeline (Step 3)
- Run inline hard-limit matcher, post-gate classifier, and mechanical-consistency audit
- On all-pass: playback validated tokens into `pending_narration`
- On exhaustion: emit fallback narration
- Emit `resolved_beat_ids` per D-06.4 (which beats from the active `SceneBrief.beats` this prose advanced)
- Honor per-turn budget per D-06.6; on `BudgetStopError` mid-pipeline, emit fallback and post `BUDGET_STOP`
- Add proper skill activation logging

### Step 6: Mechanical Consistency Audit (D-06.2)

**File:** `src/sagasmith/agents/orator/mechanics_consistency.py`

Deterministic post-generation audit. **No LLM call.**

Inputs: buffered narration text, the active `list[CheckResult]` for the turn.

Audit rules:
- **Number-token comparison:** extract integer tokens from prose. For each `CheckResult.damage` / `CheckResult.healed` / etc. that the prose appears to reference (heuristic: number appears within N words of the actor name or check kind), assert the prose number matches the result number.
- **Degree-of-success keyword check:** map degrees (`critical_success`, `success`, `failure`, `critical_failure`) to a vocabulary of accepted/forbidden keywords. Example: if degree is `success`, the prose must not contain "fails", "misses", "fumbles" near the relevant actor. The keyword tables live alongside the auditor and are unit-tested.
- **Actor-action consistency:** if `CheckResult.actor_id` is the player and the kind is `attack`, the prose must not describe the target as the attacker.

Output: `AuditResult` with `ok: bool` and `violations: list[str]` for logging. A non-OK result triggers one rewrite (counts against the two-rewrite budget shared with the post-gate).

Phase 7 may add an LLM verifier if eval shows the deterministic audit misses too many real contradictions; explicitly out of scope here.
