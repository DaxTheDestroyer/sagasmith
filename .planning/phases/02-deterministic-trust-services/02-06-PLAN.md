D-15 / PERSISTENCE_SPEC §4 turn-close write ordering (Phase 2 owns steps 1–7 only; 8–10 are Phase 7):
1. BEGIN TRANSACTION
2. Append transcript records
3. Append roll logs
4. Append provider logs (LLM call logs with secrets redacted)
5. Store applied StateDeltas
6. Append cost log entries
7. Store checkpoint reference(s)
8. COMMIT

(Note: PERSISTENCE_SPEC lists "checkpoint via SQLite saver" as step 6 and cost logs implicit; Plan 02-06 swaps cost_logs to step 6 and checkpoint_refs to step 7 because the LangGraph SQLite saver lands in Phase 4 — we record only a reference row in Phase 2.)

Turn record write is step 8-equivalent: `INSERT INTO turn_records (status='complete', ...)` within the same transaction AFTER steps 2–7 succeed. If any step raises, ROLLBACK and raise `TrustServiceError` wrapping the original.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Typed SchemaModel → SQLite | Every write goes through a repository that accepts a validated model; no raw dict INSERT paths. |
| Mid-transaction failure → caller | A failed turn-close must leave zero evidence; PERS-04 depends on this. |
| Persisted trust tables → log/export | Any export (debug dump, support bundle) must pass RedactionCanary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-28 | Tampering | Partial turn-close state | mitigate | `close_turn` wraps all 7 steps in explicit BEGIN/COMMIT; rollback on any exception; PERS-04 test proves zero turn_records after failure. |
| T-02-29 | Repudiation | Missing audit trail | mitigate | Seven tables cover transcript, rolls, provider calls, cost, state deltas, checkpoints, turn status. Every insert includes ISO timestamp. |
| T-02-30 | Information Disclosure | Secret in persisted table | mitigate | Provider log builder (Plan 02-03) redacts `safe_snippet` before persistence; table-level RedactionCanary sweep in smoke check #11 and test_redaction_over_all_tables_clean catches regressions. |
| T-02-31 | Tampering | SQL injection via user input | mitigate | All SQL uses parameterized queries; no f-string concatenation. |
| T-02-32 | Denial of Service | Unbounded transcript growth | accept | Phase 2 scope is trust-records persistence; transcript pruning is Phase 7 (memory/vault) per D-15 and the CONTEXT.md deferred list. |
| T-02-33 | Elevation of Privilege | WAL journal leak | accept | WAL journal lives next to DB file in the local campaign dir; local-first threat model does not treat local filesystem as hostile. Phase 7 vault scope will document encryption-at-rest if needed. |
