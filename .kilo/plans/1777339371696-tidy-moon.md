## Objective

Unblock Phase 5 planning for **Rules-First PF2e Vertical Slice** by generating and verifying the required UI design contract, rerun the GSD phase planner so it can produce executable `PLAN.md` files for Phase 5, then run the cross-AI review workflow to produce `05-REVIEWS.md`.

- Phase 5 includes UI scope via `TUI-07`: reveal-mode dice overlay or equivalent modal showing DC, modifier, d20 result, total, and degree.

- The recommended path is to run `/gsd-ui-phase 5`, not `/gsd-plan-phase 5 --skip-ui`, because Phase 5 includes player-facing TUI behavior.

## Execution Plan

1. Generate the Phase 5 UI design contract.
   - Run `/gsd-ui-phase 5` as the top-level next command.
   - Validate the phase through the UI workflow against the canonical `.planning/ROADMAP.md` Phase 5 entry.
   - Accept the non-blocking warnings if Phase 5 `CONTEXT.md` or `RESEARCH.md` is missing; the UI researcher may ask unresolved design questions.
   - If a Phase 5 UI-SPEC appears before execution, choose whether to update, view, or skip to checker rather than overwriting blindly.
   - Let the workflow create the Phase 5 planning directory if needed.
   - Provide user decisions if the UI researcher asks about reveal-mode dice presentation, modal/overlay behavior, keyboard handling, copy tone, and how dice details integrate with the existing Textual shell.
   - Expected output: `.planning/phases/05-rules-first-pf2e-vertical-slice/05-UI-SPEC.md`.

2. Verify and revise the UI contract through the built-in checker.
   - Allow `gsd-ui-checker` to validate the generated contract against the six UI quality dimensions.
   - Require the checker to use the generated UI-SPEC as primary input and compare against any available Phase 5 context or research.
   - If blocked, let the revision loop update only the UI-SPEC issues identified by the checker.
   - Do not force approve unless remaining items are non-blocking flags and clearly acceptable.
   - Expected final UI result: `## UI-SPEC VERIFIED` or equivalent approved status from the checker.

3. Rerun Phase 5 planning after UI-SPEC is approved.
   - Run `/gsd-plan-phase 5`.
   - Do not use `--skip-ui` unless the user explicitly chooses to accept planning without a UI contract.
   - If prompted about missing Phase 5 context, prefer continuing only if the UI-SPEC captures the necessary interaction decisions; otherwise run `/gsd-discuss-phase 5` first and then rerun planning.

4. Ensure Phase 5 planning uses the right canonical scope.
   - Requirements: `RULE-04` through `RULE-12`, `TUI-07`, and `QA-03`.
   - Specs to anchor planning: `docs/specs/PF2E_MVP_SUBSET.md`, `docs/specs/STATE_SCHEMA.md`, `docs/specs/PERSISTENCE_SPEC.md`, `docs/specs/GAME_SPEC.md`, and existing Phase 2-4 plans/summaries for DiceService, roll logs, SQLite turn records, TUI command stubs, graph checkpoints, and skills.
   - Preserve the trust-before-breadth rule: deterministic services own modifiers, DCs, damage, HP changes, action counts, degree outcomes, roll logs, and persistence.

5. Accept only a verified Phase 5 plan set.
   - Let `gsd-plan-checker` run the normal verification loop.
   - Require generated plans to cover the visible pregen sheet, deterministic checks, initiative, simple combat, action economy, position tags, roll-log auditability, reveal-mode dice UI, and PF2e mechanics tests.
   - Expected final outputs: Phase 5 `*-PLAN.md` files plus any workflow-generated research/context artifacts.

6. Run cross-AI plan review after Phase 5 plans exist.
   - Run `/gsd-review --phase 5` or `/gsd-review --phase 5 --all` as a top-level command.
   - Let the workflow detect available external AI CLIs: Gemini, Claude/the agent, Codex, CodeRabbit, OpenCode, Qwen, and Cursor.
   - Skip the current runtime's own CLI if required by `review.md` so at least one independent reviewer is used.
   - If no independent external CLI is available, install/configure at least one supported CLI before retrying.

7. Collect and synthesize review output.
   - Expected output: `.planning/phases/05-rules-first-pf2e-vertical-slice/05-REVIEWS.md`.
   - Ensure the file includes per-reviewer sections, reviewer metadata, plans reviewed, and a consensus summary.
   - Treat empty or failed reviewer output as a recorded reviewer failure, not as successful feedback.

8. Incorporate review feedback if substantive concerns are found.
   - Run `/gsd-plan-phase 5 --reviews` after `05-REVIEWS.md` exists.
   - Require revised plans to address shared HIGH/MEDIUM concerns, especially scope gaps, dependency ordering, deterministic-rule authority, TUI reveal-mode UX, replayability, persistence, and test coverage.
   - Rerun review only if the revised plan set changes materially or the first review found high-risk issues.
