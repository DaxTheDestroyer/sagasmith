## Risks / Things To Sanity-Check Before Executing

These are worth a human pass before `/gsd-execute-phase 2` because they affect downstream blast radius:

1. **`LLMRequest.response_format`** — Plan 02-03 treats this as `"text" | "json_schema"` per the provider spec. Confirm this still matches the shape Orator (Phase 6) expects; any change later forces a migration of `ProviderLogRecord`.
2. **`ProviderLogRecord` persistence** — Plan 02-06 adds `provider_logs` as a Phase 2 trust table. The spec is silent on retention; the plan defaults to "keep forever until repair commands prune." Confirm that's acceptable or specify retention now.
3. **`close_turn()` scope** — Plan 02-06 enforces steps 1–7 of `PERSISTENCE_SPEC §4` (SQLite phase) and explicitly stops before step 8 (vault writes). A future Phase 7 plan must extend, not replace, `close_turn`; the review should confirm the signature is forward-compatible.
4. **Static pricing table** — Plan 02-04 commits `pricing_table.json` to the repo. Confirm we are comfortable treating that as canonical (vs. fetching at runtime) for MVP.
5. **Opt-in live OpenRouter env var** — Plan 02-05 uses `SAGASMITH_RUN_LIVE_OPENROUTER=1` as the gate. If you want a different name or a CLI flag instead, say so now before tests pin the name.
6. **Wave 3 parallelism invariant** — The plan-checker pass flagged that 02-04 and 02-05 must not both edit `services/__init__.py` or `providers/__init__.py` during their wave. If you expect them to land serially anyway, this invariant is free safety; if you plan to parallelize, keep it in mind during execution.

None of these are blocking; each is a "confirm the locked direction is still what you want" item.
