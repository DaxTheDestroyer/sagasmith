# Plan: Safety Guard Module

## Goal

Deepen backlog item 8 by creating a Safety Guard Module that concentrates content-policy matching, pre-generation routing, inline streaming scan, post-generation scan, event construction, and retry/fallback policy behind one Safety Guard Interface.

The current code is shallow because callers and tests must know multiple partial Interfaces:

- `SafetyPreGate` owns pre-generation routing and one copy of policy synonyms/regexes.
- `SafetyPostGate` owns post-generation scanning and another copy of policy synonyms/regexes.
- `SafetyInlineMatcher` owns streaming hard-limit matching and another copy of policy synonyms/regexes.
- `content_policy_routing` owns Oracle skill routing and another copy of policy synonyms/regexes.
- `scene_rendering` owns the two-rewrite ladder, fallback narration policy, and safety event construction inline.

Deletion test result: deleting any one of the current scanners would not remove complexity; it would force policy matching and safety decisions back into Oracle, Orator, or tests. A Safety Guard Module earns Depth because deleting it would spread policy knowledge and retry/fallback rules back across multiple callers.

## Architecture Shape

Add a new top-level package:

- `src/sagasmith/safety_guard/__init__.py`
- `src/sagasmith/safety_guard/guard.py`

The Safety Guard Module Interface should be small and behavior-rich:

- Construct once from `ContentPolicy | dict | None`, optional `LLMClient`, `cheap_model`, `max_rewrites`, and fallback narration text.
- `route_intent(scene_intent)` returns allow, reroute, or block before Oracle scene generation.
- `check_stream(buffer_text)` returns a hard-limit hit for buffered Orator streaming text, or no hit.
- `scan_generated_prose(prose, attempt_index)` returns allow, retry, or fallback for completed Orator prose.
- `retry_or_fallback(...)` centralizes the two-rewrite limit for inline hits, post-gate violations, and mechanical-consistency retry decisions.
- `make_event(turn_id, kind, policy_ref, reason)` returns the player-visible `SafetyEvent` value used by state updates.

The Module Implementation should own:

- One canonical synonym/regex table.
- One compiled policy matcher for hard limits and soft limits.
- Redaction/reroute text replacement.
- Soft-limit action semantics: `fade_to_black`, `avoid_detail`, `ask_first`.
- Inline hard-limit scanning.
- LLM-backed post-generation classifier prompt and deterministic fallback scan.
- Translation from scan findings to retry/fallback decisions.

Real Seams:

- The post-generation classifier has two real Adapters: LLM-backed classification and deterministic inline fallback when no `LLMClient` exists or the LLM call fails.
- Existing public modules can remain as legacy Adapters over the new Interface while call sites migrate.

## Expected Benefits

Locality:

- Policy terms, synonyms, regexes, redaction, classifier fallback, retry/fallback limits, and event construction live in one Module.
- Safety rule changes stop requiring edits in Oracle skill logic, pre-gate code, post-gate code, inline matcher code, and Orator rewrite code.
- Safety bugs can be reproduced through one Safety Guard Interface instead of by driving multiple graph paths.

Leverage:

- Oracle scene planning, Orator streaming, post-generation QA tests, and legacy scanner Adapters all use one Safety Guard Interface.
- Existing fake and no-LLM paths get the same behavior as paid-provider paths.
- Future player-vault redline checks can reuse the same policy matcher without adding another synonym table.

Testability:

- The Interface becomes the primary test surface for SAFE-01, SAFE-02, SAFE-03, and QA-05 behavior.
- Orator tests can focus on rendering and mechanics consistency because safety policy is verified separately.
- Scene Planning tests can focus on state updates and interrupts because content-policy matching is verified separately.

## Risks And Mitigations

Risk: accidental behavior drift in existing tests.

Mitigation: keep legacy Adapter classes and run existing service, integration, regression, Oracle, and Orator tests before deleting any public names.

Risk: introducing circular imports through `services`.

Mitigation: place the Safety Guard Module in a top-level `sagasmith.safety_guard` package and import only schemas/provider protocols, not `services.__init__`.

Risk: over-broad Interface.

Mitigation: keep persistence out of Safety Guard. It should construct event values and decisions; existing graph/runtime paths can persist through the current `SafetyEventService` named class.

Risk: Orator retry/fallback policy is shared with mechanical-consistency audit, not only safety.

Mitigation: expose a generic retry/fallback helper from Safety Guard for player-facing narration retry policy, while keeping the mechanical audit Implementation in Orator.

## Non-Goals

- Do not change player-facing safety UX beyond preserving current `/pause`, `/line`, pre-gate, post-gate, and fallback behavior.
- Do not add new policy taxonomy or a moderation vendor.
- Do not persist events from the Safety Guard Module directly.
- Do not re-open ADR-0001. The plan keeps LangGraph nodes thin and leaves node Adapter responsibilities intact.
