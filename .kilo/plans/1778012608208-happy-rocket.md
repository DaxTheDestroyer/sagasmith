# Plan: Rules Turn Resolution Module

## Goal

Deepen backlog item #9 by moving one player rules turn out of `rules_lawyer_node()` and into a Rules Turn Resolution Module. The node remains the LangGraph Adapter required by ADR-0001, while the new Module owns deterministic PF2e first-slice turn resolution behind one Interface.

## Current Friction

- `src/sagasmith/agents/rules_lawyer/node.py` currently owns normalization, parsing guardrails, intent fallback, first-slice defaults, `RulesEngine` construction, `CombatEngine` construction, skill activation decisions, error wording, phase changes, and narration audit messages.
- The deterministic rules Modules already have useful Depth, but callers still need too much ordering knowledge to run one rules turn correctly.
- The deletion test says `rules_lawyer_node()` is earning workflow complexity, but the wrong Module owns it. Deleting the node would scatter the same rules-turn choreography across graph tests, TUI flows, replay, and smoke checks.
- The Interface under test is currently a graph-node dictionary shape. That lowers Locality because most rules-turn regressions are tested through node plumbing rather than the rules-turn concept.

## Target Shape

Create a new package:

- `src/sagasmith/rules_turn_resolution/__init__.py`
- `src/sagasmith/rules_turn_resolution/builder.py`

Proposed public Interface:

- `RulesTurnContext`: a frozen dataclass carrying `state`, `dice`, `cost`, optional `llm`, optional `provider_config`, and optional `skill_store`.
- `RulesTurnResult`: a frozen dataclass carrying `state_updates` and `skills_activated`.
- `resolve_rules_turn(context: RulesTurnContext) -> RulesTurnResult`.

This keeps the external Seam small: callers ask the Module to resolve the current player rules turn and receive graph-ready state updates plus skill names for the Adapter to log.

## Implementation Steps

1. Add SagaSmith domain vocabulary after acceptance.

- Update `docs/sagasmith/SAGASMITH_CONTEXT.md` with `Rules Turn Resolution` because the new Module name is a SagaSmith runtime concept, not only coding-harness vocabulary.
- Define it as the deterministic Rules Lawyer workflow for one player rules turn, owning player-intent classification, first-slice defaults, PF2e check/combat execution, rules error shaping, combat phase transitions, and narration audit messages.

2. Create the Rules Turn Resolution Module.

- Add `builder.py` with `RulesTurnContext`, `RulesTurnResult`, and `resolve_rules_turn()`.
- Add `__init__.py` exporting only the Interface.
- Follow the existing deepened Module style from `scene_planning` and `turn_plan`: plain dataclasses, pure state-in/state-out behavior, no LangGraph imports, no SQLite writes, no Textual knowledge.

3. Move rules-turn orchestration behind the new Interface.

Move these responsibilities from `rules_lawyer_node()` into `resolve_rules_turn()`:

- Raw input extraction and normalization.
- Legacy `roll perception` rejection behavior.
- Failed mechanical input detection.
- Scene context extraction from `scene_brief`.
- Deterministic intent resolution before optional LLM fallback.
- Provider model selection from `provider_config`.
- First-slice character defaulting through `make_first_slice_character()`.
- `RulesEngine` and `CombatEngine` construction.
- Action dispatch for `skill_check`, `start_combat`, `strike`, `move`, and `end_turn`.
- Combat completion phase update back to `play`.
- Damage roll narration audit message.
- Stable visible `Rules error:` wording.

Keep existing deterministic Modules (`RulesEngine`, `CombatEngine`, `intent_resolution`, `first_slice`) as internal Implementation collaborators, not new external seams.

4. Keep the node as the LangGraph Adapter.

Refactor `src/sagasmith/agents/rules_lawyer/node.py` so it only:

- Records `_call_recorder` entry when present.
- Builds a `RulesTurnContext` from `state` and `AgentServices`.
- Calls `resolve_rules_turn()`.
- Projects `RulesTurnResult.state_updates` onto the graph update dict.
- Uses `get_current_activation()` to log each returned skill name.

The node should no longer construct rules engines, parse player input, decide rules error text, or know first-slice defaults.

5. Preserve Agent Skills behavior without adding a fake Seam.

- Let the Rules Turn Resolution Module decide which skill names were activated (`skill-check-resolution` or `combat-resolution`).
- If `skill_store` is available, validate activation by attempting `load_skill()` and silently skipping missing or unauthorized skills, matching the current direct-test tolerance.
- Keep `AgentActivationLogger.set_skill()` in the node Adapter because activation logging is graph-runtime plumbing, not rules-turn Implementation.

6. Add Module-level tests.

Create `tests/rules_turn_resolution/test_builder.py` that calls `resolve_rules_turn()` directly. Cover the Interface as the test surface:

- No `pending_player_input` returns empty updates.
- Explicit skill/perception checks append exactly one `CheckResult`.
- Natural-language skill input resolves without an LLM call.
- Non-mechanical player input returns empty updates.
- Unsupported mechanical input returns stable `Rules error:` narration without rolling.
- Budget fallback returns the existing explicit-check hint.
- `start combat` creates first-slice combat state, initiative results, phase `combat`, and `combat-resolution` activation.
- `strike` consumes one action, appends one attack result, preserves damage roll audit narration when damage occurs, and returns phase `play` when combat completes.
- Invalid strike range/target preserves existing checks and returns visible rules error.
- `move` and `end turn` update combat state through the same Interface.
- Input state is not mutated.

7. Thin existing node tests instead of duplicating coverage.

- Keep a small `rules_lawyer_node()` contract test proving it delegates, remains pure, records call order, and logs returned skills inside graph activation.
- Migrate detailed behavior assertions from `tests/agents/test_rules_lawyer_phase5.py` into the new Module tests.
- Preserve existing integration tests that prove graph/TUI/replay paths still cross the same Seam.

8. Re-run focused and regression checks.

Run focused tests first:

- `uv run pytest tests/rules_turn_resolution/test_builder.py`
- `uv run pytest tests/agents/test_rules_lawyer_phase5.py tests/agents/test_node_contracts.py tests/integration/test_intent_resolution_flow.py`
- `uv run pytest tests/services/test_rules_engine.py tests/services/test_combat_engine.py tests/services/test_intent_resolution.py tests/agents/rules_lawyer/test_intent_to_proposal.py`

Then run repository gates appropriate for this refactor:

- `uv run ruff check .`
- `uv run ruff format --check .`
- `uv run pyright`
- `uv run pytest`

9. Update architecture backlog status after implementation.

- Mark item #9 in `.kilo/ARCHITECTURE-DEEPENING-BACKLOG.md` as implemented.
- Note the new Module path, the retained Adapter shim, and the new direct Interface tests.

## Expected Benefits

- Locality: rules-turn choreography moves from a graph node into one Module, so future changes to first-slice action routing, error wording, combat phase transitions, or narration audit messages happen in one place.
- Leverage: graph runtime, TUI smoke, replay, and future rules inputs can all cross the same Rules Turn Resolution Interface instead of duplicating rules-turn ordering knowledge.
- Testability: rules-turn behavior becomes directly testable without LangGraph dictionaries as the primary test surface, while node tests shrink to Adapter behavior.
- AI-navigability: future agents and maintainers can search for the SagaSmith concept `Rules Turn Resolution` and land on the Module that owns one player rules turn.

## Non-Goals

- Do not redesign PF2e math in `RulesEngine` or `CombatEngine`.
- Do not introduce a new Adapter Interface for dice, rules, or combat; one adapter would be a hypothetical Seam here.
- Do not expand first-slice rules scope beyond the existing skill checks, start combat, strike, move, and end turn behavior.
- Do not reopen ADR-0001; this plan preserves LangGraph nodes as Adapters and keeps deterministic work below the orchestration layer.
