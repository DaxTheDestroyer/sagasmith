# Neo4j Friction Analysis for AI-TTRPG

**Mode:** Research / decision-support plan (no code changes implied yet).
**Context:** `docs/specs/GAME_SPEC.md` §3.5 (Archivist uses a graph for entity/
canon); `docs/WISHLIST.md` §5.1 (Kuzu noted as deferred alternative).
**Goal of this document:** Give the user a full picture of why Neo4j was
flagged as a friction point so they can make an informed choice about the
storage backend before planning starts.

---

## 1. TL;DR

Neo4j is the right shape (a graph DB with Cypher) but the wrong size for
this project. The app is:

- **Single-player**, local-first, installed by hobbyists.
- Intended to be **easy to share** ("others to enjoy it").
- Likely distributed via `pip install` / a single binary.

Neo4j is a **server-based JVM database** designed for multi-user,
high-concurrency workloads. Almost every property that makes it good in
production (separate server, auth, clustering, heap tuning) is a cost in a
single-user local app. The three alternatives worth weighing are:

1. **Kuzu** — embedded, Cypher-compatible, MIT-licensed (closest drop-in).
2. **SQLite-only** — model the "graph" as adjacency tables; reuse the DB
   already required for transcripts and profile.
3. **NetworkX + JSON/SQLite persistence** — pure Python in-process graph;
   smallest dependency footprint.

If the project is primarily a **learning exercise that should be
frictionless to install**, Neo4j is the weakest of the four. If the user
specifically wants to learn Cypher / production graph DB ergonomics, Neo4j
or Kuzu is the right choice — with Kuzu winning on distribution.

---

## 2. Why Neo4j was chosen originally (steelman)

Before listing problems, record why it showed up in the spec:

- **Mature Cypher ecosystem.** Lots of docs, tutorials, Stack Overflow
  answers, and graph-algorithm libraries.
- **Rich query language.** Pattern matching, variable-length paths, and
  `MATCH ... MERGE` are genuinely nice for entity/canon work like "find
  every NPC who has ever met the PC and is still alive."
- **First-class tooling.** Neo4j Browser / Bloom let you visually inspect
  the graph during development — useful when debugging canon drift.
- **Well-known brand.** Reviewers/contributors recognize it.
- **It's a great thing to learn.** If the point is skill-building, working
  with a real graph DB is more transferable than faking it in SQL.

Keep this in mind: "friction" is about distribution and operational cost,
not about whether Neo4j is a good database. It is.

## 3. The friction points, categorized

### 3.1 Installation and distribution friction (biggest issue)

This is the category that was specifically called out in `WISHLIST.md`.

- **Not pip-installable.** Neo4j is a JVM application. A Python package
  cannot `pip install neo4j-community`. The user gets the `neo4j` *driver*
  (a client) but must install the *server* separately.
- **Three supported install paths, each with friction:**
  1. **Neo4j Desktop** — GUI app, multi-hundred-MB download, requires
     account sign-up for some flows, GUI-only on Windows/Mac.
  2. **Docker** — assumes the user has Docker Desktop, which on Windows
     means WSL2 + ~2 GB of dependencies for a toy app.
  3. **Tarball/zip + bundled JRE** — works, but you're shipping a JVM
     inside your Python app, and now you own the upgrade path.
- **Onboarding tax.** The spec promises "Install → `ttrpg init` → add
  OpenRouter key → onboarding → start campaign, all in one session"
  (GAME_SPEC §11). A Neo4j install step breaks that flow. Every dependency
  that isn't `pip install`-shaped costs real adoption.
- **Platform variance.** Install steps differ per OS. You will end up
  writing three sets of docs and three bug-report triage paths. A hobby
  project cannot afford this.

### 3.2 Resource and performance friction

- **JVM memory baseline.** Neo4j defaults to ~512 MB–1 GB heap plus page
  cache; on a laptop that already has the TUI, the LLM client, LanceDB,
  and SQLite open, this is non-trivial.
- **Cold-start time.** JVM warmup + Neo4j bootstrap is on the order of
  10–30 s. That's a visible delay the user sees before `ttrpg play` works.
- **Background process.** Neo4j runs as a daemon. If the user force-quits
  the TUI, a Neo4j process may be left behind, ports may be held, or the
  database may need recovery on next boot.
- **Workload mismatch.** The actual graph size for a campaign is small —
  probably tens of thousands of nodes at most over a long campaign.
  Neo4j's performance envelope (millions of nodes, many concurrent
  writers) is wasted here, while its startup and memory costs are paid in
  full.

### 3.3 Operational complexity friction

- **Auth and ports.** Neo4j defaults to requiring a username/password and
  binds bolt://localhost:7687. Even for a local-only app you must:
  - generate/store credentials (or disable auth, which ships a surprise
    security footgun if the user ever opens the port),
  - pick a port and handle collisions,
  - add a connection-health probe before the first query.
- **Migrations.** Neo4j storage format has broken between major versions
  in the past. Each upgrade may require `neo4j-admin dump` / `load`, which
  is fine for a DBA and terrible for a hobbyist who just wants their
  campaign back.
- **Backups.** "Copy my campaign" needs to be a single file copy for a
  normal user. With SQLite that's trivial. With Neo4j it's a dump command
  or a full server shutdown and directory copy.
- **Crash semantics.** If the machine loses power mid-turn, SQLite's WAL
  recovers cleanly. Neo4j is also durable, but its recovery path (store
  lock files, schema check) is more surfaces to break on a consumer
  machine you don't control.

### 3.4 Licensing friction

- Neo4j Community Edition is **GPLv3**. For a personal training project
  published under your own terms, that's usually fine — but:
  - It means anyone distributing a modified fork of your app *together
    with* Neo4j must comply with GPL.
  - Enterprise features (causal clustering, hot backups, certain index
    types) are **commercial-only**, not available to you.
  - If you ever want to ship a commercial or permissively-licensed
    binary, Neo4j is a blocker; Kuzu (MIT) or SQLite (public-domain-ish)
    is not.
- This is usually a non-issue for a hobby project but is the sort of
  surprise people hit six months in.

### 3.5 Developer-experience friction inside *this* codebase

- **Two databases to keep consistent.** The spec already requires SQLite
  (profiles, transcripts, checkpoints) and LanceDB (embeddings). Adding
  Neo4j makes **three** stores that must be written transactionally to
  keep canon coherent (§3.5: "Every named entity resolves to exactly one
  graph node"). Three-store consistency with one of them being a separate
  server process is notably harder than two-store with both embedded.
- **Testing.** Unit tests either need a Neo4j test container (slow, flaky
  in CI) or an in-memory fake. Kuzu and SQLite both have trivially
  spin-up-and-tear-down semantics for tests.
- **Local dev setup for contributors.** Any contributor — or any future
  "you" in 6 months — has to reinstall Neo4j to run the test suite. This
  is the single biggest contributor-churn risk for a hobby project.

## 4. How much graph power does this app actually need?

This is the most important question, because it changes which alternative
is best.

The graph-shaped queries implied by the spec are:

- "Has the PC met this NPC before, and if so, what was the disposition?"
- "What relationships exist between these two NPCs?" (faction, family,
  debt)
- "Show me every callback seeded in arcs the player has touched."
- "Entity resolution: does 'the dwarven innkeeper' match any existing
  NPC node within fuzzy-match distance?"
- "List all places reachable from the current location within N hops of
  the travel graph."

Observations:

- **All of these are expressible in SQL** with a handful of join tables
  and recursive CTEs. They're not convenient, but they work.
- **None of them require variable-length path queries at scale.** The
  hops involved are small (1–3 typically).
- **The killer feature of a graph DB here is ergonomics**, not
  performance. Cypher just reads nicer than the SQL equivalent.
- **LanceDB already handles the fuzzy semantic side** of entity
  resolution. The graph is only doing the structured side.

This means the honest answer is: **this app doesn't need a graph DB for
performance reasons. It wants one for modeling ergonomics.** That reframes
the Neo4j-vs-alternatives question entirely.

## 5. The realistic alternatives, scored

| Option | Install friction | Cypher? | File-portable? | License | Risk |
|---|---|---|---|---|---|
| **Neo4j (status quo)** | High (JVM + server) | Yes | No (needs dump) | GPLv3 / commercial | Low (battle-tested) |
| **Kuzu** | Very low (pip / wheel) | Yes (subset) | Yes (single dir) | MIT | Medium (young; some Cypher gaps) |
| **SQLite adjacency tables** | None (stdlib) | No | Yes | PD | Low (verbose queries) |
| **NetworkX + JSON/SQLite snapshot** | Very low (pure Python) | No | Yes | BSD | Medium (you own durability) |
| **Apache AGE / Postgres** | High (Postgres server) | Yes | No | Apache 2 | Low (but wrong shape) |
| **Memgraph** | High (server) | Yes | No | BSL | Low (wrong shape) |

Two honorable mentions the user didn't ask about but are worth noting:

- **DuckDB + property-graph extension.** DuckDB is embedded and has
  growing graph-ish support; not a Cypher engine but has path queries.
  Low install friction, good analytical side-effects.
- **FalkorDB (née RedisGraph).** Cypher on Redis. Faster than Neo4j for
  small graphs but still needs a server and the project has had
  turbulent stewardship.

### 5.1 Why Kuzu is the leading alternative

- Ships as a Python wheel. `pip install kuzu` — no separate server.
- Cypher-compatible for the queries this app needs.
- Single-file / single-directory database, backup = `cp -r`.
- MIT-licensed: no downstream constraints.
- Actively developed and used in production at some shops, so it is no
  longer "too new to trust."
- The adapter pattern called out in WISHLIST §5.1 makes swapping feasible
  if you start with Kuzu and discover a gap.

**The real risk with Kuzu** is feature drift vs. Neo4j (some APOC
procedures, some index types, some Cypher clauses). For this app's query
set, none of the missing pieces appear to matter, but this should be
spot-checked by writing the 5–8 "real" Cypher queries the Archivist will
issue and running them against Kuzu before committing.

### 5.2 Why SQLite-only is the dark-horse contender

- You already have SQLite for transcripts/profile/checkpoints.
- Zero new dependencies. Zero install friction.
- Backups = "copy the .db file." Matches player intuition.
- Consistency is trivial: one file, one transaction, done.
- The cost is query ergonomics: recursive CTEs are ugly and you give up
  Cypher's pattern matching.
- This is the choice if **ease of distribution is the #1 goal** and the
  user doesn't particularly care about learning graph-DB tooling.

### 5.3 Why NetworkX + persistence is the simplest possible start

- Pure Python, in-process.
- Enormously expressive for graph algorithms.
- Persistence is manual: snapshot to JSON or SQLite on checkpoint.
- Falls over past a few hundred thousand nodes (not a concern here).
- Good fit if you want to prove the modeling first and pick a "real" DB
  later.

## 6. Recommendation framing (decision inputs, not a decision)

The right choice depends on which of these the user values more, so the
plan is explicitly *not* making the call. The axes:

1. **"I want this to install cleanly for non-technical friends."**
   → Kuzu or SQLite-only. Neo4j is wrong.
2. **"I want to learn real production graph-DB ergonomics."**
   → Kuzu (Cypher, embedded) gives 80% of the Neo4j learning at 10% of
     the friction. Only pick Neo4j if you specifically want to learn
     Neo4j *operations*.
3. **"I want the smallest possible dependency surface."**
   → SQLite-only or NetworkX. Both let you ship a project with one
     database file and no server ever.
4. **"I want to keep Neo4j and just document the friction."**
   → Legitimate, but budget for contributor churn and install-support
     tickets. At minimum, wrap Neo4j in an adapter so a swap later is
     mechanical.

## 7. Suggested next steps (when the user is ready to decide)

These are *planning* next steps, not implementation. Nothing here writes
code yet.

1. **Enumerate the concrete Cypher queries** the Archivist will need
   (probably 8–12). This is a 30-minute exercise and makes every
   alternative comparable.
2. **Check each query against Kuzu's current Cypher support matrix** to
   validate it as a real drop-in.
3. **Decide on an adapter boundary** so whichever backend is picked, the
   rest of the app talks to a `GraphStore` interface. This is already
   foreshadowed by the WISHLIST §5.1 entry ("adapter pattern allows
   swap") and GAME_SPEC §3.5.
4. **Pick the backend** based on §6's axes.
5. **Update `GAME_SPEC.md` §3.5 and `WISHLIST.md` §5.1** to reflect the
   decision, so the next planning pass starts from a clean contract.

## 8. Open questions for the user

Before converting this analysis into an implementation plan, the following
need explicit answers:

- **Q1. Primary audience priority.** Is "frictionless install for friends"
  a hard requirement, or a nice-to-have? If hard → rule Neo4j out now.
- **Q2. Learning goal.** Is learning Cypher / graph DB tooling one of the
  explicit goals of the project? If yes → Kuzu stays in, SQLite-only
  drops out.
- **Q3. Distribution model.** Is the plan `pip install ttrpg`, a
  PyInstaller single binary, source-clone-and-run, or Docker? This
  changes which friction points are real and which are theoretical.
- **Q4. Willingness to adapter-wrap.** Are you comfortable paying a small
  abstraction cost up front (a `GraphStore` Protocol) to keep the backend
  swappable? If yes, the backend choice becomes reversible and lower-
  stakes.
- **Q5. Cross-platform scope.** Windows + Mac + Linux, or just your own
  platform? Neo4j's install pain is especially bad on Windows without
  WSL2 or Docker.

Answers to these convert this analysis into a concrete implementation
plan.
