# Plan: Agent Skills Execution Module

## Goal

Deepen backlog item #10 by creating an Agent Skills Execution Module that concentrates Agent Skills runtime behavior behind one Interface. `SkillStore` keeps owning filesystem discovery and validation, while the new Module owns per-agent authorization, instruction loading, activation recording, and invocation of the selected Python Implementation.

This preserves ADR-0001: LangGraph nodes remain Adapters, Agent Skills remain portable filesystem assets, and deterministic SagaSmith behavior stays below graph orchestration.

## Current Friction

- `src/sagasmith/skills_adapter/store.py` and `loader.py` have useful Depth for discovery, validation, and `load_skill`, but runtime callers use them unevenly.
- `src/sagasmith/agents/*/node.py` calls `get_current_activation().set_skill(...)` directly, so graph activation plumbing leaks into each node Adapter.
- `src/sagasmith/scene_planning/builder.py` checks `skill_store.find(...)` locally and imports Oracle skill Implementations directly.
- `src/sagasmith/rules_turn_resolution/builder.py` imports `load_skill` directly only to authorize/log skill names; it still carries local activation logic.
- `src/sagasmith/agents/archivist/node.py` has a hard-coded skill-name tuple and logs skills by checking `store.find(...)`, while `src/sagasmith/turn_plan/builder.py` directly invokes Archivist skill Implementations without returning skill activations.
- `src/sagasmith/agents/orator/node.py` logs `scene-rendering` manually and invokes `render_scene(...)` directly.

Deletion test result: deleting the scattered activation snippets would not remove complexity; authorization, instruction loading, duplicate suppression, and activation logging would reappear in each node or deepened Module. That means the runtime skill path deserves a deeper Module with one Seam.

## Target Shape

Add the Module in the existing Agent Skills package:

- `src/sagasmith/skills_adapter/execution.py`

Proposed public Interface:

- `AgentSkillExecution`: bound to one `agent_name` and optional `SkillStore`.
- `AgentSkillExecution.catalog()` or `catalog_text()`: returns the same compact catalog used for prompt injection.
- `AgentSkillExecution.load(skill_name)`: authorizes and loads full skill instructions through `load_skill`.
- `AgentSkillExecution.activate(skill_name)`: authorizes, loads instructions, records activation locally, and sets the current graph activation when present.
- `AgentSkillExecution.run(skill_name, implementation, *args, **kwargs)`: activates the skill and invokes the supplied Python Implementation.
- `AgentSkillExecution.activated_names`: deterministic, duplicate-free tuple of skill names activated during this bound execution object.

Use `ParamSpec`/`TypeVar` so `run(...)` preserves the wrapped Implementation return type without forcing all skills into one fake payload type.

This keeps the Interface small. Callers still own domain-specific arguments for each skill Implementation, while the Module owns the repeated Agent Skills runtime rules.

## Key Semantics

- If no `SkillStore` is configured, `activate(...)` and `run(...)` should execute provider-free test paths without crashing and should not record an activation. This preserves current direct-test tolerance.
- If a `SkillStore` is configured, `activate(...)` should use `load_skill(...)` so not-found and unauthorized behavior is centralized.
- Missing or unauthorized skills should not be recorded as activations. For production paths, the default store scan should still fail loudly on invalid `SKILL.md` files at bootstrap.
- Duplicate activation names within one `AgentSkillExecution` object should be suppressed while preserving first-activation order.
- `run(...)` should not swallow Implementation exceptions. Graph activation outcome remains owned by `AgentActivationLogger`.
- Import `get_current_activation` lazily inside `execution.py` so importing `SkillStore` remains lightweight.

## Expected Benefits

- Locality: skill authorization, instruction loading, duplicate suppression, and activation logging move from scattered nodes and workflow Modules into one Module.
- Leverage: Oracle, Rules Lawyer, Orator, Archivist, Onboarding, tests, and future Agent Skills all cross the same execution Seam.
- Testability: the Agent Skills runtime rules become testable through their Interface instead of through graph-node side effects.
- AI-navigability: future maintainers can search for `AgentSkillExecution` or `Agent Skills Execution` and find the Module that owns runtime skill behavior.

## Non-Goals

- Do not redesign `SkillStore` discovery or frontmatter validation.
- Do not dynamically import Python Implementations from `SKILL.md` names. Current skill Implementations have heterogeneous signatures, and several Agent Skills are implemented by deeper deterministic Modules rather than one `logic.py` function.
- Do not add a second skill store Adapter without a real second Adapter need.
- Do not change the `agent_skill_log` table shape in this refactor. The current graph activation row can continue recording the last activated skill for a node turn; a multi-skill audit schema can be a separate persistence change if required.
- Do not reopen ADR-0001. This plan strengthens the ADR by making the LangGraph-to-Agent-Skills glue deeper and more consistent.
