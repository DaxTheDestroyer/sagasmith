# Phase 7 – Discuss Phase Plan

**Goal:** Produce a `CONTEXT.md` that captures implementation decisions for downstream agents (researcher, planner). The context must include:
- Clear phase boundary statement
- Locked requirements (if SPEC.md exists)
- All decisions made during discussion
- Full canonical references with relative paths
- Codebase context (reusable assets, patterns)
- Deferred ideas and reviewed todos
- Discussion log (audit trail)

## Workflow Overview

1. **Initialize**
   - Resolve phase metadata (`phase_dir`, `padded_phase`, etc.) via `gsd-sdk query init.phase-op`.
   - Detect `DISCUSS_MODE` (`discuss`, `assumptions`, `power`).

2. **Validate Blocking Anti‑Patterns**
   - Look for `.continue-here.md` → parse `blocking` rows.
   - Answer the three anti‑pattern questions inline before proceeding.

3. **Check for SPEC.md**
   - If present, load `<locked_requirements>` and treat them as pre‑answered.
   - Ensure downstream agents know these are locked.

4. **Load Prior Context**
   - Read `.planning/PROJECT.md`, `.planning/REQUIREMENTS.md`, `.planning/STATE.md`.
   - Read all prior `*-CONTEXT.md` files and extract `<prior_decisions>`.
   - Load any spike/sketch findings (`spike-findings-*/SKILL.md`).

5. **Cross‑Reference Todos**
   - Use `todo.match-phase` to surface relevant pending todos.
   - Offer to fold selected todos into this phase’s scope (store as `<folded_todos>` / `<reviewed_todos>`).

6. **Scout Codebase**
   - Prefer existing `.planning/codebase/*.md` maps.
   - If none, grep for phase‑relevant terms and read 3‑5 most relevant files.
   - Build internal `<codebase_context>` for later annotation.

7. **Analyze Phase & Identify Gray Areas**
   - Using `prior_decisions` and `codebase_context`, generate concrete gray‑area questions.
   - Annotate each with:
     - Relevant prior decision (if any)
     - Code context / pattern reference
   - Skip areas already decided or validated by spikes.

8. **Present Gray Areas**
   - Show phase boundary, carried‑forward decisions, and the list of gray areas.
   - Use a multi‑select question for the user to pick which areas to discuss.
   - In `--auto`/`--all` mode, auto‑select all areas and skip selection UI.

9. **Advisor Research (optional)**
   - If `ADVISOR_MODE` is active, spawn parallel `gsd-assumptions-analyzer` agents for each selected area.
   - Synthesize comparison tables + rationale, rewrite for non‑technical user if needed.
   - Store tables for later discussion.

10. **Discuss Selected Areas**
    - **Interactive mode:** Ask focused questions (max 4) per area, with options annotated by code context.
    - **Batch mode (`--batch`):** Ask 2‑5 related questions in one turn.
    - **Text mode (`--text`):** Present numbered list for user to select.
    - Record each answer, follow‑up if ambiguity remains.
    - Auto‑mode (`--auto`) selects recommended option and logs the choice.

11. **Incremental Checkpointing**
    - After each area, write `${phase_dir}/${padded_phase}-DISCUSS-CHECKPOINT.json` with:
      - Completed areas, remaining areas, decisions, deferred ideas, canonical refs.
    - Allows graceful resume if the session is interrupted.

12. **Write CONTEXT.md**
    - Create `${phase_dir}/${padded_phase}-CONTEXT.md` using the standard template.
    - Sections required:
      - `<domain>` – phase boundary statement.
      - `<spec_lock>` – if SPEC.md existed.
      - `<decisions>` – list decisions (D‑01, D‑02, …) grouped by logical area.
      - `<canonical_refs>` – **MANDATORY** full relative paths to all specs, ADRs, design docs.
      - `<code_context>` – reusable assets, patterns, integration points.
      - `<specifics>` – product‑specific references or “I want it like X”.
      - `<deferred>` – out‑of‑scope ideas.
    - Ensure `<canonical_refs>` is present even if empty (state “No external specs …”).

13. **Write DISCUSSION‑LOG.md**
    - Full audit trail of questions, options, answers, and any “chat more” notes.
    - Not consumed by downstream agents.

14. **Git Commit & State Update**
    - Commit `CONTEXT.md` and `DISCUSSION‑LOG.md` with a descriptive message.
    - Record session info in `.planning/STATE.md`.

15. **Confirm & Next Steps**
    - Print a concise summary with the path to the newly created `CONTEXT.md`.
    - Offer shortcuts: `/gsd-plan-phase ${PHASE}` to continue, view deferred ideas, or edit the context.

## Success Criteria

- Phase validated against roadmap; no re‑asking of decided questions.
- All gray areas identified, presented, and discussed until user satisfaction.
- `CONTEXT.md` fully populated with decisions, canonical refs, and code insights.
- DISCUSSION‑LOG.md captures the entire Q&A audit trail.
- STATE.md updated; next phase (`plan-phase`) can be launched automatically (if `--auto` or config set).
- User knows the exact next command to proceed.
