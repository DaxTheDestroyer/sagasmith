# Phase 8 Implementation Plan: Retcon, Repair, and Release Hardening

## Goal

Implement Phase 8 after Phase 7 completes: safe completed-turn retcon behavior plus MVP no-paid-call smoke and release gates.

Phase 8 must satisfy `QA-01`, `QA-02`, `QA-08`, and `QA-09`:

- User can retcon a selected completed turn after explicit confirmation.
- Retconned turns remain audit-retained but are excluded from canonical replay, summaries, memory packets, vault rebuilds, player-vault projection, derived indices, `/recap`, and future gameplay context by default.
- Developer can run a no-paid-call MVP smoke path covering install/entrypoint, init, configure, onboard, skill challenge, simple combat, quit, and resume.

## Implementation Steps

### 1. Persistence And Canonical Read Contract (`08-01`)

- Add migration `0008_retcon_audit.sql` for `retconned` turn status, `retcon_audit`, and `vault_write_audit`.
- Extend persistence schemas with `RetconAuditRecord`, `VaultWriteAuditRecord`, and `TurnRecord.status == "retconned"` support.
- Add repository helpers for recent eligible completed turns, affected suffixes, marking turns retconned, retcon audits, vault-write audits, and canonical transcript reads.
- Route Archivist transcript context through the canonical transcript helper so retconned rows are excluded by default.
- Record successful vault writes during turn close to support retcon impact preview.

### 2. MVP Smoke And Release Gate (`08-04`)

- Add `run_mvp_smoke()` with named no-paid-call checks:
  - `mvp.install_entrypoint`
  - `mvp.init`
  - `mvp.configure_fake`
  - `mvp.onboard`
  - `mvp.play_skill_challenge`
  - `mvp.play_simple_combat`
  - `mvp.quit`
  - `mvp.resume`
- Extend `sagasmith smoke --mode mvp` and include at least one shell-level `uv run sagasmith smoke --mode mvp` test.
- Add `secret-scan` and `release-gate` Make targets without replacing existing quality targets.
- `release-gate` should run lint, format check, typecheck, tests, MVP smoke, and secret scan.

### 3. Retcon Service And Runtime Integration (`08-02`)

- Create `src/sagasmith/persistence/retcon.py` with `RetconService`, preview/result models, candidates, and `RetconBlockedError`.
- Preview must list affected turn ids, prior checkpoint id, transcript/mechanics/vault impacts, and exact token `RETCON {turn_id}`.
- Confirmation must recompute preview, require exact token, mark affected canonical suffix turns as `retconned`, append audit record, rewind graph state to the prior checkpoint, rebuild derived layers from canonical sources, and resync player vault.
- If safe checkpoint data is missing or invalid, block with repair guidance and do not guess or partially rewrite canon.
- If derived rebuild or player-vault sync fails after status/audit commit, return actionable repair guidance while canonical SQL exclusion already protects future context.

### 4. TUI `/retcon` Flow (`08-03`)

- Replace the Phase 8 stub in `src/sagasmith/tui/commands/control.py`.
- `/retcon` with no args lists recent eligible completed turns and does not silently target the latest turn.
- `/retcon <turn_id>` renders preview, affected turns, vault outputs, effects, and typed confirmation instructions.
- `/retcon <turn_id> RETCON <turn_id>` confirms through `app.graph_runtime.confirm_retcon(...)`.
- Success messaging must be concise, spoiler-safe, and avoid removed transcript/vault body details.
- Blocked messaging must include repair guidance without exposing GM-only content, secrets, or removed canon details.
- If needed, add a small app helper to resync TUI state after retcon without exiting the session.

## Key Risks And Controls

- Canon leakage risk: enforce retcon exclusion through shared repository/query helpers, not scattered filters.
- Audit loss risk: mark status and append audit metadata; do not destructively delete trust records as the primary mechanism.
- Unsafe rollback risk: require a prior safe checkpoint and block when it is missing or invalid.
- Player-facing spoiler risk: retcon preview/completion messages may include ids/counts/effects, but not removed canon details, GM-only content, or provider/secret data.
- Release false confidence risk: combine in-process smoke observability with shell-level CLI entrypoint proof and release-blocking secret scan.

- Phase 8 plan summaries exist for `08-01` through `08-04`.
- `QA-01`, `QA-02`, `QA-08`, and `QA-09` are implemented and covered by tests.
- Retconned content is excluded from canonical future gameplay context and derived rebuilds by default.
- `/retcon` is explicit, reversible only through safe checkpoint semantics, and does not force session exit after success.
- `make release-gate` passes before Phase 8 is marked complete.
