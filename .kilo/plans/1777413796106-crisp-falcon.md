# Phase 7 Plan: Memory, Vault, and Resume Differentiator

**Goal:** User gets persistent campaign memory, spoiler-safe Obsidian vault projection, recap, repair commands, and later-process resume
**Depends on:** Phase 6 (AI GM Story Loop) complete
**Requirements:** CLI-04, PERS-03, PERS-05, PERS-06, VAULT-01 through VAULT-10, AI-11, TUI-08, QA-06
**Success Criteria** (what must be TRUE):
1. User can open a player vault containing Obsidian-compatible sessions, NPCs, locations, factions, items, quests, callbacks, lore, index, and log pages with no GM-only leakage.
2. The system atomically writes validated master-vault pages, projects `player_known` and `foreshadowed` content safely, and strips GM-only frontmatter and comments from player projections.
3. Archivist can resolve named entities, detect canon conflicts, and assemble bounded MemoryPackets from permitted search/retrieval sources.
4. User can run `/recap`, quit from the TUI, resume after a later process start, and see prior NPCs, quests, events, and transcript context recalled correctly.
5. User can run vault validation, player-vault sync, and derived-index rebuild commands when repair warnings or corruption are detected.

**Planned Sub-Plans:** 6

## 07-01: Vault I/O Foundation + Entity Resolution

**Scope:**
- Create `sagasmith.memory.vault` package with `VaultPage`, page-type schemas (NPC, Location, Faction, Item, Quest, Callback, Session, Lore), and YAML frontmatter helpers.
- Implement `AtomicVaultWriter` with tempfile write + `os.replace()` + post-write YAML validation.
- Implement `EntityResolver` with slug generation (§4.1 VAULT_SCHEMA), alias search across pages of same type, and LanceDB vector similarity placeholder (interface only, implementation deferred to 07-03).
- Add migration 0007_vault_meta_table.sql to track `vault_page_versions` (filename, last_modified, schema_version, visibility, canonical_id). This enables incremental sync and rebuild tracking.
- Seed the master vault directory (`~/.ttrpg/vault/<campaign_id>/`) and player vault directory (`<campaign_dir>/player_vault/`) on campaign initialization (CLI already creates player_vault dir; Phase 7 upgrades it to vault structure).
- Unit tests: slug generation, collision handling (`_2` suffix), alias matching round trips, YAML validation rejects invalid frontmatter, atomic write partial-file protection, `vault_page_versions` tracking.

**Outcome:** Working Obsidian-compatible vault write path with entity resolution guarantees; master vault directory structure and version tracking established.

## 07-02: Player Vault Projection + Visibility Sync

**Scope:**
- Implement `PlayerVaultProjector` that iterates master vault pages and applies visibility rules:
  - `gm_only` → skip entirely
  - `foreshadowed` → write stub (id, type, name, aliases only; body replaced with `*Unknown — you have heard this name but know little more.*`)
  - `player_known` → write full page with GM-only fields stripped (frontmatter fields: `secrets`, `gm_notes`, any `gm_*` prefixed) and `<!-- gm: … -->` HTML comment blocks removed from body.
- Atomic per-page write using `AtomicVaultWriter`.
- Post-sync: regenerate `index.md` (auto-generated world overview) and append to `log.md` (append-only event log) per VAULT_SCHEMA §5.8-5.9.
- Wire sync into `close_turn` AFTER step 7 (SQLite commit) as step 8.5; if sync fails, mark turn `needs_vault_repair` and raise visible TUI warning (PERS-05).
- Add `vault_sync_report` table (migration 0008) to record last_sync_success, last_error, pages_written, pages_skipped.
- Unit tests: full projection fixture (master → player diff), GM-only content leakage detection, `index.md` generation matches expected NPC/location/quest lists, `log.md` appends correctly, atomic tempfile protection during forced failure.

**Outcome:** Spoiler-safe player vault projection producing readable Obsidian artifacts; sync failure surfaces repair warning without losing completed turn.

## 07-03: Derived Indices: FTS5 + LanceDB + NetworkX

**Scope:**
- **SQLite FTS5:** Create `vault_fts` table with triggers on `vault_page_versions` changes. Initial full population from master vault bodies. Incremental updates during sync (only changed pages). Basic full-text search API: `search_fts(query, limit=5)` → list of matching canonical page IDs.
- **LanceDB (stub):** Initialize `LanceDBClient` interface with `embed_and_upsert(page_id, text)` and `semantic_search(query, limit)` methods — both are no-ops returning empty results in Phase 7 (D-01). The interface and directory (`~/.ttrpg/lancedb/`) are created so post-MVP activation requires no surgery.
- **NetworkX graph:** Build directed multigraph from wikilinks (`[[wikilinks]]`) and explicit frontmatter edges (`connects_to`, `factions`, `related_entities`). Persist as `graph_version` under `~/.ttrpg/graph/` using `networkx.write_gpickle`. Load on startup; recompute only if `vault_page_versions` indicate changes since last graph write.
- Rebuild command infrastructure: `sagasmith vault rebuild --all` regenerates all three derived layers from master vault alone. Unit tests verify rebuild idempotency and incremental sync correctness.
- Migration 0009 creates `derived_index_versions` table to store last_built_at timestamps and checksums for FTS/LanceDB/NetworkX.

**Outcome:** Rebuildable derived read layers supporting future MemoryPacket retrieval; corruption recovery via `vault rebuild`.

## 07-04: Archivist Skill - Turn-Close Persistence + Vault Upsert

**Scope:**
- Implement `archivist-skills.md` §2.7 `turn-close-persistence` as the primary skill wired into `archivist_node`. This skill orchestrates:
  1. Assemble `TurnCloseBundle` from state (transcript entries, roll results, provider logs, state deltas, cost logs, checkpoint refs, turn record).
  2. Call `close_turn(conn, bundle)` from Phase 2 (PERS-01/02/04 already exists) to write SQLite transactionally.
  3. On SQLite success, iterate changed vault pages from `StateDelta`s that target `vault_` namespace and write each via `AtomicVaultWriter`.
  4. Upsert `vault_page_versions` tracking for each written page.
  5. Trigger derived index updates (FTS5 upserts; LanceDB stub no-op; NetworkX incremental graph merge) — these may be async/fire-and-forget but completion is before turn marked fully done to keep repair guarantees.
  6. If any vault write fails (IOError, validation), mark turn status `needs_vault_repair` and raise `TrustServiceError`; do **not** commit derived indices.
  7. Return `PersistenceReport` (success flag, pages_written, errors, needs_repair).
- Update `archivist_node` return value to include `last_persistence_report` and update `session_state.turn_count`.
- Extend `StateDelta` schema with `target: Literal["vault_npc", "vault_location", "vault_faction", "vault_item", "vault_quest", "vault_callback", "vault_session", "vault_lore"]` and `vault_payload: dict[str, Any]` to authorize vault writes.
- Unit tests: successful turn-close writes vault + FTS + NetworkX in correct order (LanceDB no-op); vault IO failure propagates to `needs_vault_repair`; invalid vault frontmatter is rejected before file write.

**Outcome:** Deterministic turn-close persistence with atomic SQLite + vault writes; repair flag surface for derived-index failures.

## 07-05: Memory Packet Assembly + Rolling Summary + Recap Command

**Scope:**
- Replace Phase 6 `assemble_memory_packet_stub` with full retrieval implementation per `archivist-skills.md` §2.1 `memory-packet-assembly`.
- Retrieval pipeline (token-bounded, default cap 2048):
  1. **Exact match**: Recent 3 transcript turns (already present).
  2. **Semantic (stub)**: LanceDB `semantic_search` call is a no-op returning empty in Phase 7 (D-01); interface call is preserved so post-MVP activation requires no changes to this pipeline.
  3. **Graph neighborhood**: For each entity, fetch 1-hop wikilink neighbors from NetworkX (max 5 total).
  4. **Callbacks**: Query open callbacks matching location/NPC/quest via `callback-reachability-query` heuristic (max 3).
  5. **Summaries**: Include rolling summary from `rolling_summary` session state field (written by `rolling-summary-update`; empty string if no scene has closed yet).
- Enforce token cap across `summary + recent_turns`; drop lower-priority buckets if over.
- Implement `archivist-skills.md` §2.8 `rolling-summary-update` skill (D-08):
  - Triggered at scene close events: when Oracle replans or all active beat IDs resolve.
  - Makes one LLM call (`DeterministicFakeClient`-compatible) to summarize completed scene beats + transcript entries into a bounded text blob (max 512 tokens).
  - Stores result in `session_state.rolling_summary`; written to SQLite as a blob column in migration 0009 (see Cross-Phase section).
  - Token cap on output enforced before storing; truncates to 512 tokens to keep MemoryPacket within bounds.
- Implement `/recap` command (replaces stub in `control.py`) that:
  - Reads `rolling_summary` from session state (produced by `rolling-summary-update`).
  - Supplements with last 3–5 transcript rows for the current scene.
  - Includes current scene, active quests, known NPCs, location, clock.
  - Displays in narration area with clear section headers.
- Unit tests: packet respects token cap on long context, includes expected entities from retrieval, rolling summary updates only at scene boundary (not per-turn), recap output includes required sections.

**Outcome:** Bounded-context memory feeding Oracle/Orator with live rolling summary; functional `/recap` with zero additional LLM cost at recap time.

## 07-06: Visibility Promotion + Canon Conflict Detection + Repair/TUI Commands

**Scope:**
- **Visibility promotion** (`archivist-skills.md` §2.5): Deterministic heuristic tracking:
  - When an entity first appears in Orator narration (entity ID appears in narration text), promote `gm_only` → `foreshadowed` if not already visible.
  - When entity is explicitly named by NPC or discovered in-scene (narrated "you meet X" or "you find Y"), promote to `player_known`.
  - Implement `promote_visibility_if_needed(page_id, turn_transcript)` called from post-narration archivist. Record promotion event in `log.md`.
- **Canon conflict detection** (`archivist-skills.md` §2.6):
  - Before writing vault updates, compare incoming `vault_payload` against current page frontmatter.
  - If conflicting top-level fields (e.g., `status`, `disposition_to_pc`, `location_current`) differ beyond a white-list of safe mutations (e.g., `status: alive` → `status: dead` is allowed; `name` change is not), emit `CanonConflict` record to `pending_conflicts` in `SagaState`.
  - Oracle consumes conflicts on next planning turn and must explicitly acknowledge resolution; if unresolved, writes are rejected and `Orator` narrates "something feels wrong" without overwriting.
- **Repair commands** (CLI-04):
  - `sagasmith vault validate`: scan master vault all pages, report invalid YAML, missing required fields, broken wikilinks. Exit code non-zero on errors.
  - `sagasmith vault sync`: force re-run player vault projection from scratch. Useful when derived indices indicate mismatch.
  - `sagasmith vault rebuild`: rebuild FTS5, LanceDB, NetworkX from master vault. Option `--force` clears and rebuilds everything; default is incremental based on `vault_page_versions`.
- **Session page authoring** (`archivist-skills.md` §2.9 `session-page-authoring`):
  - Runs once per session on quit (before player-vault-sync), generating a `session/session-NNN.md` vault page.
  - Makes one LLM call to summarize session beats, notable NPCs encountered, locations visited, quests advanced, and significant rolls — using the rolling summary as input to avoid re-reading full transcript.
  - Writes the page atomically via `AtomicVaultWriter` and upserts `vault_page_versions`. The session page is always `player_known` visibility.
  - Wired into the quit path after `turn-close-persistence` and before `player-vault-sync`.
- **TUI quit/resume** (TUI-08):
  - Wire `app.action_quit()` to run archivist node one last time if a turn is in `narrated` state (i.e., Orator finished but Archivist persistence not yet run due to crash). This ensures the turn is checkpointed and vault written before exit.
  - Add `/quit` command to gracefully stop the TUI and run the same cleanup.
  - On next `sagasmith play`, resume works because final checkpoint exists (GRAPH-05 already guarantees this; TUI-08 wires the UI exit path).
- **QA-06 regression:** Add test that runs a short 3-turn fixture, inspects every player vault file, and asserts zero occurrences of `gm_notes`, `secrets`, or `<!-- gm:` patterns.

**Outcome:** Full differentiator package: spoiler-safe memory, conflict-aware canon guard, repair tooling, clean quit/resume.

## Cross-Phase Dependencies and Notes

**Schema migrations sequence (all Phase 7 migrations):**
- 0007_vault_meta_table.sql — `vault_page_versions` (07-01)
- 0008_vault_sync_report.sql — `vault_sync_report` (07-02)
- 0009_derived_index_versions.sql — `derived_index_versions` + `rolling_summary` blob column (07-03/07-05)
- 0010_fts5_schema.sql — FTS5 virtual table and triggers (07-03)
- 0011_networkx_graph_checkpoint.sql — `graph_checkpoints` table storing NetworkX pickle checksum (07-03)

**Skills registration:** Each Archivist skill (turn-close-persistence, entity-resolution, vault-page-upsert, visibility-promotion, canon-conflict-detection, memory-packet-assembly, rolling-summary-update, session-page-authoring, player-vault-sync) gets a `SKILL.md` under `src/sagasmith/agents/archivist/skills/` and is loaded by `SkillStore` with `first_slice=True`. The LanceDB stub interface is registered with `first_slice=False` (no paid calls in Phase 7; interface present for post-MVP activation).

**TUI command wiring:** `/recap` and `/quit` added to `sagasmith.tui.commands.control`; `/vault` subcommand group added to CLI with `validate`, `sync`, `rebuild` subcommands.

**Determinism boundaries:** All vault I/O, entity resolution (slug/alias), visibility promotion, and conflict detection are deterministic. LanceDB `semantic_search` is a no-op stub in Phase 7 — always returns empty, always deterministic.

**Safety:** No LLM-generated memory content is vault-authoritative. All vault writes are mediated through `StateDelta` validation and Archivist deterministic services. GM-only leakage regression QA-06 asserts zero exposure across 50-turn fixture.

**Cost:** Zero LLM calls in normal vault write/rebuild paths. `rolling-summary-update` and `session-page-authoring` each make one LLM call at scene/session close; both support `DeterministicFakeClient` so no paid calls in tests.
