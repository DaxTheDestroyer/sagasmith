# Plan: Per-Agent Skills Catalog — Starting with ArchivistAgent

**Status:** Draft, awaiting confirmation  
**Created:** 2026-04-26  
**Prior art:** `docs/specs/ADR-0001-orchestration-and-skills.md` (LangGraph + Agent Skills decision), `docs/specs/GAME_SPEC.md` §3, `docs/specs/VAULT_SCHEMA.md`

---

## 1. Goal

Produce the first of six **skills catalog design artifacts** — one per MVP agent plus one for shared supporting services — that enumerate the capabilities each agent will eventually ship as `SKILL.md` packages in the codebase. The catalogs are pre-implementation design documents; they are **not** the `SKILL.md` files themselves. They are the spec that each `SKILL.md` will be authored against.

This plan covers:

- The **structure and conventions** for the catalog pass as a whole (applies to all six agents).
- The **complete first-pass catalog for ArchivistAgent**, pre-populated in enough detail that execution is primarily transcription with light tuning.
- **Open decisions** requiring user input before execution begins.
- **Acceptance criteria** for the Archivist catalog artifact.
- The **sequencing and rough scope** for the remaining five catalogs.


### 3.2 Rationale for one file per agent (not a single `AGENT_SKILLS.md`)

Considered: a single monolithic `docs/specs/AGENT_SKILLS.md` holding all six catalogs. Rejected because:

- Archivist alone runs ~10 skills × 6 fields × ~5 lines each ≈ 300+ lines. Six agents at that density is 1,800+ lines in one file, which breaks both human review and future agent-specific PR scoping.
- Per-agent files let us evolve individual catalogs independently without merge conflicts.
- When implementation begins, each agent's PR naturally touches its own catalog, giving one-to-one traceability.
- A single `docs/specs/agents/README.md` provides the index/cross-ref view that the monolith would have given.

### 3.3 Template-file anchor

The six-field template lives in `docs/specs/agents/README.md` (see §4). Every catalog references the README's template rather than restating it, keeping catalogs focused on their skills.

## 4. The six-field skill template (restated)

Agreed in the design conversation before plan mode. Lives in `README.md` for reference.

| Field | Purpose |
|---|---|
| **Name** | `kebab-case`, max 64 chars, becomes the `name:` frontmatter of the eventual `SKILL.md`. Stable across docs. |
| **Purpose** | One sentence: what the agent can *do* because of this skill. Feeds the `description:` frontmatter. |
| **Inputs → Outputs** | The data contract, in plain language. Cites relevant typed state fields (from `graph/state.py`) where applicable. |
| **Implementation surface** | One of: `deterministic` (pure code, possibly with scripts/), `prompted` (LLM reasoning is the core), `hybrid` (LLM + deterministic pipeline), `tool-call` (delegation to another agent or external service). This drives what `scripts/`, `references/`, and `assets/` the eventual `SKILL.md` bundles. |
| **Key dependencies** | Other skills, services (DiceService, SafetyGuard, etc.), storage layers (SQLite, LanceDB, vault), or schemas. |
| **Success signal** | One concrete observable that tells us the skill works. Seeds at least one eval fixture. |

Each skill entry may optionally include a **"Notes / open questions"** field for design-phase ambiguity that isn't resolvable without implementation experience.

## 5. ArchivistAgent — pre-computed skills catalog

Distilled from `GAME_SPEC.md` §3.5 and the whole of `VAULT_SCHEMA.md`. Ten skills, grouped into read-side, write-side, and lifecycle.

### 5.1 Scope context

Archivist owns:

- The master vault (`~/.ttrpg/vault/<campaign_id>/`).
- The player vault sync (projection to `<campaign_dir>/`).
- SQLite transcripts, roll logs, checkpoints (shared with LangGraph's `SqliteSaver`).
- LanceDB embeddings index.
- NetworkX graph (in-memory, rebuilt from vault).
- FTS5 index (derived from vault).

Archivist's "skills" are heavily biased toward deterministic pipelines with LLM adjudication at specific tie-break points. Several skills bundle significant `scripts/` content; a few are genuinely prose-heavy LLM work (summary generation, canon conflict interpretation).

### 5.2 Skill enumeration (ten skills)

Ordered from read-side → write-side → lifecycle.

---

#### Skill 1: `memory-packet-assembly`

- **Purpose:** Given a scene context and player input, assemble a token-bounded `MemoryPacket` that the graph passes to Oracle/Orator for the current turn.
- **Inputs → Outputs:** `(scene_context, player_input, budget_tokens)` → `MemoryPacket` containing ranked retrievals from FTS5 (exact-phrase hits), LanceDB (semantic neighbors), NetworkX (k-hop entity neighborhood), and the active rolling summary.
- **Implementation surface:** `hybrid` — deterministic multi-source retrieval + ranker; LLM used only for over-budget compression.
- **Key dependencies:** SQLite FTS5 index, LanceDB, NetworkX graph, rolling-summary store, `PlayerProfile.pacing` (affects how aggressive compression is).
- **Success signal:** On a 50-turn fixture session, MemoryPacket never exceeds the configured token cap; semantic-recall for NPC reintroduction after 10 sessions correctly retrieves the canonical page (see GAME_SPEC §3.5 acceptance criterion).

---

#### Skill 2: `callback-reachability-query`

- **Purpose:** Given the current scene and recent player actions, return the set of open callbacks that could plausibly be activated this turn, scored by narrative fit.
- **Inputs → Outputs:** `(current_scene, recent_turns, open_callbacks_ledger)` → `[(callback_id, fit_score, activation_hints)]`.
- **Implementation surface:** `hybrid` — deterministic prefiltering (location/NPC/quest overlap) + LLM scoring of narrative fit for top candidates.
- **Key dependencies:** `callbacks/` pages in master vault, NetworkX graph for relationship paths, `quests/` pages for active-quest context.
- **Success signal:** Oracle's "payoff vs. seed" decision receives non-empty candidate lists on ≥ 80% of scenes in a 10-session fixture; callback seeded in session 1 becomes reachable when its trigger conditions match in a later session.

---

#### Skill 3: `entity-resolution`

- **Purpose:** Before creating a new vault page, resolve whether a named entity already exists in canon, enforcing the "one entity, one page" invariant.
- **Inputs → Outputs:** `(incoming_name, entity_type, contextual_description)` → `(match_existing_page_id | create_new)`.
- **Implementation surface:** `hybrid` — three-step deterministic pipeline (slug match → alias match → LanceDB vector similarity at ≥ 0.92 cosine threshold) with LLM adjudication on vector matches between 0.85–0.92 (ambiguous band).
- **Key dependencies:** VAULT_SCHEMA §4.1 slug rules, VAULT_SCHEMA §8 resolution procedure, LanceDB, all page-type aliases.
- **Success signal:** Entity-resolution precision ≥ 0.95 on the fixture suite (GAME_SPEC §3.5 acceptance criterion); zero duplicate NPC pages in a 50-turn regression.

---

#### Skill 4: `vault-page-upsert`

- **Purpose:** Atomically write or update a master-vault page with schema-valid frontmatter, handling slug collisions and preserving existing GM-only content.
- **Inputs → Outputs:** `(page_type, frontmatter, body, gm_only_blocks)` → `(vault_path, created|updated)`.
- **Implementation surface:** `deterministic`. Pure script. Validates frontmatter against the page-type schema (VAULT_SCHEMA §5), resolves slug via §4.1 rules, appends `_2/_3` on collision, writes via tempfile + `os.replace()`.
- **Key dependencies:** VAULT_SCHEMA §4 (filenames), §5 (page schemas), `schema-validation` cross-cutting skill.
- **Success signal:** No partial writes on induced I/O failure (tempfile rollback works); schema-invalid frontmatter is rejected before disk write; collision suffixing produces `_2.md` correctly.

> **Notes / open questions:** Should this skill be bundled *inside* `entity-resolution` (since the two are rarely invoked separately) or remain standalone? Argument for keeping separate: `entity-resolution` is read-oriented (produces a decision); `vault-page-upsert` is write-oriented (executes the decision). Argument for combining: the LLM never calls upsert without first resolving. **Current plan: keep separate** — the test surfaces and failure modes are different. Revisit during implementation if the split creates friction.

---

#### Skill 5: `visibility-promotion`

- **Purpose:** Promote a vault page's `visibility` field along the one-way progression `gm_only → foreshadowed → player_known` based on in-turn events (drops, reveals, direct encounters).
- **Inputs → Outputs:** `(page_id, turn_transcript, current_visibility)` → `new_visibility | unchanged`.
- **Implementation surface:** `prompted`. Pure LLM reasoning: did the PC actually encounter this entity in the turn? Was the name dropped in conversation? Does this rise to "foreshadowed"? The rules are written, but the judgment is interpretive.
- **Key dependencies:** VAULT_SCHEMA §6 visibility-state table, the turn transcript, the page's current visibility and type (NPCs and locations promote differently from callbacks).
- **Success signal:** On a 20-turn fixture with labeled expected promotions, agent-determined promotions agree with the gold set ≥ 0.9 of the time; no demotions occur.

---

#### Skill 6: `canon-conflict-detection`

- **Purpose:** Compare facts the player asserted in their input against canon in the vault; emit `CanonConflict` events that Oracle surfaces in the next `SceneBrief`.
- **Inputs → Outputs:** `(player_input, turn_transcript, relevant_vault_pages)` → `[CanonConflict]` (possibly empty).
- **Implementation surface:** `prompted`. LLM identifies factual assertions in the player's prose and compares them to canonical fields (species, role, location, status). Distinguishes three cases: retcon intent, PC misbelief, narrator error — each handled differently.
- **Key dependencies:** `entity-retrieval` (to fetch the vault pages being asserted about), `MemoryPacket` content for the active turn.
- **Success signal:** On a fixture of 15 crafted conflict scenarios, agent correctly categorizes (retcon / misbelief / narrator-error) ≥ 0.8 of the time, and never silently overwrites canon.

> **Notes / open questions:** Where exactly does the `CanonConflict` event land in the LangGraph state? Proposal: a dedicated `pending_conflicts: list[CanonConflict]` field on the graph state, consumed and cleared by Oracle's next `scene-brief-composition` skill invocation. Defer to graph-state design work.

---

#### Skill 7: `turn-close-persistence`

- **Purpose:** Execute the bundle of write-side operations that must run at every turn close, as a single transactional unit.
- **Inputs → Outputs:** `(turn_transcript, state_deltas, new_or_changed_pages)` → `persistence_report` (success / partial-rollback).
- **Implementation surface:** `deterministic`. Bundles: transcript append to SQLite, roll-log append, embedding upsert to LanceDB for any changed page, NetworkX graph refresh. Atomicity discipline: SQLite writes in one transaction; LanceDB and NetworkX are rebuildable, so their failures are logged and don't block the turn.
- **Key dependencies:** SQLite, LanceDB, NetworkX, `vault-page-upsert` (must have completed before this skill runs).
- **Success signal:** After a forced crash between SQLite commit and LanceDB upsert, `ttrpg vault rebuild` restores LanceDB to consistency; transcript and roll log are always complete for every turn that reached checkpoint.

---

#### Skill 8: `rolling-summary-update`

- **Purpose:** Every N turns (or at scene boundary), produce a short canonical summary of what has happened; stored as a derived artifact that feeds future `MemoryPacket`s.
- **Inputs → Outputs:** `(recent_turns_window, prior_summary, canon_pages_touched)` → `updated_summary` (bounded length).
- **Implementation surface:** `prompted`. LLM composes the summary with strict constraints: canonical facts only (not player hypotheticals, not rumor). Pulls constraints from the turns themselves and from vault `status` and `visibility` fields.
- **Key dependencies:** Session transcripts, vault pages, `PlayerProfile.pacing` (length calibration).
- **Success signal:** Manual review of 10 generated summaries: all factual claims trace back to canonical vault pages or confirmed turn events; no hallucinated facts; length within configured bound.

> **Notes / open questions:** What is the trigger cadence? Proposal: per scene (Oracle emits a scene-close event) rather than every N turns — matches narrative structure better. Alternative: both, with a cheap per-N-turns incremental summary and a full per-scene canonical summary. Defer to Oracle's scene-close contract definition.

---

#### Skill 9: `session-page-authoring`

- **Purpose:** At session end, produce the `sessions/session_NNN.md` page — summary, beats, NPCs encountered, quests opened/closed, callbacks seeded/paid, rolls table.
- **Inputs → Outputs:** `(session_transcript, session_metadata, canon_changes_this_session)` → `session_page` (schema-valid per VAULT_SCHEMA §5.7).
- **Implementation surface:** `hybrid`. LLM composes the summary and beats; deterministic code assembles the frontmatter (IDs, dates, encountered lists) and the rolls table from the roll log.
- **Key dependencies:** SQLite transcript and roll log for the session, vault pages touched this session, VAULT_SCHEMA §5.7 schema.
- **Success signal:** Generated session page passes schema validation; beats list matches the OracleAgent-declared beats from the session; rolls table reconciles exactly with the dice log.

---

#### Skill 10: `player-vault-sync`

- **Purpose:** After every turn close, project the master vault to the player vault — stripping `gm_only` pages, stubbing `foreshadowed` pages, removing GM-only frontmatter fields and `<!-- gm: -->` inline blocks. Regenerate `index.md` and append to `log.md`.
- **Inputs → Outputs:** `(master_vault_path, player_vault_path, changes_since_last_sync)` → `sync_report` (pages written, stubbed, skipped).
- **Implementation surface:** `deterministic`. Pure script. Implements VAULT_SCHEMA §2.3 sync contract, §4.4 stripping rules, §7 inline comment stripping, §5.8/§5.9 index and log regeneration.
- **Key dependencies:** VAULT_SCHEMA §2, §4.4, §5.8, §5.9, §6, §7; atomic-write primitive shared with `vault-page-upsert`.
- **Success signal:** After 5 fixture sessions, opening the player vault in Obsidian shows zero GM-only content (GAME_SPEC §3.5 acceptance criterion); all `[[wikilinks]]` in player vault resolve; YAML frontmatter parses without error; player vault diff against master vault shows only expected strips/stubs.

---

#### Skill 11: `master-vault-unlock` *(lifecycle — invoked once per campaign)*

- **Purpose:** On `ttrpg vault unlock` at end of campaign, produce the "director's cut" — copy full master to player vault location, disable stripping, generate `meta/epilogue.md` summarizing open callbacks, faction dispositions, NPC fates, and Oracle-planned vs. actual arc.
- **Inputs → Outputs:** `(campaign_id, player_vault_destination)` → `unlocked_vault_path`, `epilogue_page`.
- **Implementation surface:** `hybrid`. Deterministic copy + strip-disable; LLM composes the epilogue narrative.
- **Key dependencies:** Full master vault contents, `meta/campaign_seed.md` (Oracle's original plan), all callbacks' final statuses.
- **Success signal:** Unlocked vault is a valid Obsidian vault (all wikilinks resolve, frontmatter parses); epilogue correctly lists every open and paid-off callback; players report the artifact feels like a satisfying wiki of their campaign (qualitative; no automated test).

---

### 5.3 Count check and granularity discipline

Eleven skills total (ten gameplay + one lifecycle). Matches the ADR's guidance of "6–10 per agent, plus lifecycle extras." Each skill has:

- A distinct invocation pattern (when it runs in the turn flow).
- A distinct failure mode.
- A distinct eval fixture.

### 5.4 Cross-cutting skills Archivist depends on but does NOT own

These will be catalogued later under `docs/specs/agents/services-capabilities.md`, not in `archivist-skills.md`. Mentioned here only as dependency signals:

- **`schema-validation`** — used by `vault-page-upsert` and `session-page-authoring`. Likely lives in the shared `sagasmith/skills/` root per ADR-0001 directory layout.
- **`safety-redline-check`** — optionally invoked by `player-vault-sync` as a defence-in-depth check that no redlined content leaks into the player vault. SafetyGuard owns the implementation; Archivist is a consumer.
- **Atomic-write primitive** — not a skill; a plain utility shared with `vault-page-upsert` and `player-vault-sync`.

### 5.5 Consumers of Archivist skills (informational, to anticipate other catalogs)

- Oracle consumes: `memory-packet-assembly`, `callback-reachability-query`, `entity-retrieval` (implicit — fetched via packet).
- Orator consumes: `memory-packet-assembly` (the packet is already assembled by the time Orator runs).
- Rules Lawyer consumes: minimal — usually gets the character sheet and combat state directly from typed graph state; may query `entity-retrieval` for target-NPC stat block lookups.
- Onboarding consumes: none (Archivist is not active during onboarding).

## 6. Resolved design decisions

All four open decisions were resolved in a single checkpoint before execution (user accepted all recommendations, 2026-04-26). The resolutions are binding conventions for this catalog and all subsequent ones; they are captured in `docs/specs/agents/README.md` during execution.

### 6.1 One file per agent ✅ accepted

Per-agent catalogs live under `docs/specs/agents/<agent>-skills.md` with a folder index at `docs/specs/agents/README.md`. No monolithic catalog file. Reasoning in §3.2.

### 6.2 Per-skill `Notes / open questions` field ✅ accepted

Individual skill entries may include a `Notes / open questions` sub-section when design-phase ambiguity warrants it. Keeps context with the skill rather than forcing a separate document-level section.

### 6.3 Deterministic skills still get `SKILL.md` files (Option A) ✅ accepted

Even pure-deterministic capabilities (`vault-page-upsert`, `turn-close-persistence`, `player-vault-sync`, and their equivalents in other agents) are catalogued as skills and will eventually ship as `SKILL.md` packages with `scripts/` carrying the code. The LLM-gating overhead at turn-close is negligible, and keeping the skill catalog as the single index of agent capability preserves discoverability. Revisit if activation cost proves measurable during implementation.

This decision is a binding pattern for all six catalogs, not just Archivist.

### 6.4 Services catalog uses the same template, filename `services-capabilities.md` ✅ accepted

IntentResolver, SafetyGuard, CostGovernor, DiceService, and cross-cutting skills all use the same six-field template as agent skills. The filename `services-capabilities.md` stands — the naming emphasizes that these provide capabilities consumed by agents, rather than being first-class LangGraph-node agents themselves.

## 8. Sequencing for the remaining catalogs

After Archivist ships, each subsequent catalog follows the same plan-then-execute pattern. Rough order and anticipated skill count:

| # | Agent | Rationale for this order | Anticipated skill count |
|---|---|---|---|
| 1 | **Archivist** | Data foundation. Every other agent's skills reference memory/canon contracts. | 11 (this plan) |
| 2 | **Rules Lawyer** | Mostly deterministic; quick pass. Clarifies the DiceService boundary before Oracle depends on it. | 8–10 (strike, skill check, saving throw, spell, movement/positioning, condition-application, initiative, encounter math, deg-of-success, retcon-aware replay) |
| 3 | **Oracle** | Heavy LLM work; easier to scope once Archivist and Rules Lawyer contracts are clear. | 10–14 (scene-brief composition, hook generation, callback seeding, encounter design, beat planning, NPC inline creation, content-policy routing, pacing calibration, player-choice branching, re-plan on skip) |
| 4 | **Orator** | Consumes other agents' outputs. Smallest LLM surface — mostly rendering. | 6–8 (reveal narration, hidden narration, auto narration, fade-to-black, tone matching, late-roll pause, recap rendering) |
| 5 | **Onboarding** | Bounded by fixed nine-phase flow; refit of the earlier draft. | 8–10 (restructured from the pre-plan-mode draft) |
| 6 | **Services** | Cross-cutting; most content already scaffolded by the other five. | 8–12 (IntentResolver, SafetyGuard pre/post, CostGovernor warn/stop, DiceService seeded roll + replay, schema-validation, safety-redline-check) |

Total anticipated skill catalog: **~55–70 skills across six documents**. Each catalog will get its own plan-mode pass to pre-compute the content the way this plan did for Archivist.

## 9. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Skill granularity turns out wrong (too many / too few) during implementation | Medium | Low | Catalogs are pre-implementation design docs. Merging or splitting a skill before authoring its `SKILL.md` is cheap. |
| Cross-cutting skills are duplicated across agents | Medium | Low | `services-capabilities.md` is the canonical home. Agent catalogs reference by name only; they don't restate. |
| Open decisions in §6 get resolved wrong | Low | Medium | Explicit checkpoint before execution. Recommendations provided; user can override per decision. |
| Someone authors a `SKILL.md` before its catalog entry is finalized | Low | Medium | Convention: no `SKILL.md` ships until the corresponding catalog entry is committed to `main`. Enforced by code review. |
| A skill described here doesn't match what the eventual implementation needs | Medium | Low | Catalog can be amended in place; it is a living design doc, not a contract. PR discipline requires catalog updates accompany skill changes. |
